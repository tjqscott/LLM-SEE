## 1. Introduction

This document outlines the comprehensive software requirements, deep architectural constraints, and functional targets for the development and refinement of the **Mule APIkit** framework. Designed to bridge the gap between API design and runtime implementation, APIkit serves as a foundational orchestration tool for building RESTful and SOAP web services natively within the Mule Enterprise Service Bus (ESB) ecosystem.

### 1.1 The Project at a Glance

Mule APIkit is an integration framework that empowers developers to build Mule APIs rapidly by enforcing a strict "design-first" architecture. Historically, developers manually wired HTTP listeners, choice routers, and exception handling blocks to construct an API. This bottom-up approach frequently led to drift between the implemented code and the documented API contract. APIkit reverses this paradigm: the API contract (written in RAML) becomes the executable source of truth.

The high-level goals for this phase of APIkit development are:
a. To engineer a mathematically precise HTTP routing engine that strictly adheres to REST principles, guaranteeing the return of accurate HTTP status codes (400, 404, 405, 415) when the incoming request violates the structural contract defined in the RAML specification.
b. To perfect the internal RAML Abstract Syntax Tree (AST) parser, ensuring it can dynamically resolve file inclusions, parse complex JSON/XML schemas, and construct an accurate runtime representation of the API.
c. To provide a flawless code generation experience (the Scaffolder) that translates the parsed RAML AST into functional, well-formatted, and non-duplicated Mule XML configuration files (`mule-config.xml`), drastically reducing manual boilerplate.
d. To optimize the developer bootstrapping experience by streamlining the underlying Maven Archetypes, improving JVM build performance, resolving hardcoded variables, and providing out-of-the-box example projects.

### 1.2 Definitions, acronyms, and abbreviations.

To ensure clarity throughout this document, the following technical terms are defined:

* **Archetype**: A Maven project templating toolkit (`maven-archetype-plugin`). APIkit utilizes an archetype to generate the initial folder structure, `pom.xml`, and core XML configurations for new API projects based on user input parameters.
* **Console**: An embedded, web-based graphical interface generated by APIkit (similar to Swagger UI) that allows developers to visualize the API contract and execute mock requests locally against the running Mule application.
* **Flow**: A sequence of message processors in the Mule ESB that executes integration logic. APIkit generates one main routing flow and multiple backend implementation flows.
* **RAML**: RESTful API Modeling Language. A YAML-based markup language for describing RESTful APIs. It defines resources, methods, query parameters, URI parameters, and expected payload schemas.
* **Scaffolder**: The internal Java engine within APIkit that reads the parsed RAML definition, utilizes templating engines (e.g., FreeMarker or Velocity), and outputs the corresponding XML-based Mule flows.

## 2. Requirements description.

### 2.1 Mule APIkit and its environment.

Mule APIkit operates as a dual-phase framework within the Mule ESB and Anypoint Studio environment: it functions first at *design time*, and subsequently at *runtime*.

**Design Time (Scaffolding Phase):**
A developer writes a RAML specification detailing the API contract. The APIkit Scaffolder reads this `.yaml` file, validates the syntax, and generates a foundational `mule-config.xml` file. This generated XML contains a central HTTP listener, an `<apikit:router>` component, and a series of stubbed implementation flows (one for every resource-method pair defined in the RAML).

**Runtime (Execution Phase):**
When the Mule ESB is deployed, the `<apikit:router>` acts as the central API gateway. It intercepts incoming HTTP requests from the Mule HTTP Connector. The router inspects the `HTTP_METHOD` and `HTTP_REQUEST_PATH` properties of the incoming Mule Message, validates the headers and payloads against the loaded RAML contract in memory, and then dynamically routes the message to the corresponding implementation flow (e.g., routing a `GET` request on `/users` to a flow named `get:/users:api-config`).

### 2.2 Product functions: Parsing and Scaffolding.

The core computational functionalities of the APIkit system are categorized as follows:

**RAML Parsing and Specification Management**

* **Decoupled Parser Architecture:** To ensure the parser can be maintained, versioned, and utilized by other tools independently of the Mule routing engine, the architecture team must decouple the codebase and create a separate project specifically for the Java RAML parser.
* **Modular File Resolution:** RAML supports modular design via the `!include` directive. The Java parser must be engineered to successfully locate and include another YAML file (or JSON schema) from the local filesystem or classpath. The Scaffolder must rely on this parser to find included files in the YAML to support complex, multi-file API definitions.
* **Specification Accuracy & Defaults:** The parser must strictly adhere to the RAML specification standards. During parsing, the system must recognize when default query parameters are declared in the RAML and ensure they are actively considered and injected into the Mule Message if the client omits them. Furthermore, the parser must cleanly handle optional metadata; it must not fail if the `baseUri` is undefined, and RAML files created via wizards should not inject a `baseUri` or `version` by default unless explicitly provided by the user.
* **YAML Validation:** Generated YAML files must conform to standard conventions. RAML files created should default to a `.yaml` extension. Furthermore, files generated with the Maven archetype must have the correct global tag set to `tag:raml.org`.

**Scaffolding Engine (Code Generation)**

* **Deterministic Flow Generation:** The Scaffolder translates the AST into XML. It must be highly deterministic, resolving critical bugs where it is not generating flows properly or is duplicating flows when re-run against an updated RAML file. Furthermore, to support multiple APIs in a single application, the Scaffolder should dynamically add the name of the `api-config` to the generated flow names to prevent namespace collisions.
* **XML Formatting and Whitespace:** Generated XML code must be structurally sound and human-readable. The Scaffolder must ensure that generated flows have the correct indentation, fixing issues where creating an XML file from scratch results in poor formatting, and ensuring correct carriage returns at the end of lines.
* **Configuration Integrity:** When scaffolding, the engine must set the complete path for the resource rather than truncating nested paths. It must also avoid injecting invalid special characters into the XML when a resource references an external schema. Finally, when generating a `mule-config` file from scratch, it must accurately link the `config-ref` attribute in the router element.

## 3. Other considerations.

### 3.1 HTTP Compliance, Content Negotiation, and Exception Strategy

APIkit must act as a strict, mathematically precise enforcer of REST and HTTP standards. The internal `<apikit:router>` must be hardened to automatically intercept invalid requests and execute a pre-configured Exception Strategy that returns exact HTTP status codes:

* **405 Method Not Allowed:** The router must return the correct response code when a client attempts a method that is not defined in the RAML (e.g., a member without a POST method retrieves the wrong code when trying to POST, or a wrong response code is returned when no method is defined at all).
* **400 Bad Request:** The routing engine must execute deep validation of query and URI parameters. It must return a "Bad Request" (400) when required parameters are not sent, rather than failing silently or causing a backend NullPointerException.
* **404 Not Found:** The system must ensure that deleting a non-existing member retrieves the appropriate error code (e.g., 404), rather than a generic 500 server error, fixing the issue where a non-existing member retrieves an empty string.
* **415 Unsupported Media Type / Content Negotiation:** The router must accurately parse incoming MIME types. Setting the `Content-Type` to `application/x-www-form-urlencoded` must be processed correctly into a Mule Map payload, rather than retrieving an "unsupported media type" error. Additionally, the system must ensure that the payload retrieved is successfully converted to XML when the client's `Accept` header demands it.

### 3.2 Maven Archetype Architecture and Build Performance

The developer experience relies heavily on the `maven-archetype-plugin` to scaffold project repositories automatically.

* **Archetype Accuracy:** The `mvn apikit:create` command must intelligently modify existing projects. The archetype must ensure that the config file created is automatically included in the overarching `mule-config.xml` file, and that the necessary Spring/Mule XML namespace imports are added to the header.
* **Variable Injection:** The archetype template must dynamically inject environment variables rather than hardcoded values. The default port in the scaffolder generated HTTP endpoint should be dynamically linked to the `${http.port}` property, not hardcoded to `8080`, and the archetype must explicitly add this port configuration to the HTTP endpoint XML.
* **Build Optimization:** Project builds generated by the Maven archetype currently suffer from low performance. The build engineering team must investigate the generated `pom.xml` configurations, optimize plugin executions, and create a dedicated script to populate APIkit dependencies (archetype and maven plugin) reliably and quickly across local caches.

### 3.3 Developer Console and Local Testing

To support test-driven API development, APIkit embeds a lightweight web server that hosts an interactive API Console.

* **Console Accessibility:** The built-in testing console must be highly visible to the developer. The system must explicitly list the set of console URLs in the startup logs when Mule boots up with an APIkit project.
* **Routing Fixes:** The internal HTTP routing to the console assets (HTML, CSS, JS) must be robust. The system must fix the routing bug where users cannot access the console if their configured inbound-endpoint address terminates with a trailing slash (`/`).
* **Diagnostic Logging:** When developers make mistakes in their API design, the tool must guide them. The Scaffolder must improve its internal logging output when a YAML file is not structurally valid, providing precise line numbers for syntax errors. Furthermore, RAML files that contain resource definitions but possess no implementation flows should actively lead to a compiler warning during the build process.

### 3.4 Process requirements and EE Repository Separation

To streamline open-source distribution, respect licensing boundaries, and ensure clean enterprise management, the release engineering team must meticulously separate APIkit artifacts.

The APIkit framework is intended to be available to Community Edition (CE) users as well as Enterprise Edition (EE) users. Therefore, the release engineers must explicitly remove APIKit from the strict EE Repository. Consequently, the `pom.xml` templates generated by the Maven archetype must be updated to remove EE repositories from the pom file, ensuring that community developers do not encounter HTTP 401 Unauthorized errors when attempting to download Maven dependencies.

Finally, to assist developers in learning the framework, the APIkit Maven Archetype should provide a flag or interactive option for creating an out-of-the-box example project (e.g., an example "leagues" API), giving developers a fully functional reference implementation immediately upon project generation.
