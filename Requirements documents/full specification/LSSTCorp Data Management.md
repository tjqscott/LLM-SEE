## 1. Introduction

This document delineates the comprehensive software requirements, deep architectural specifications, and strategic performance targets for the initial instantiation of the Data Management (DM) subsystem for the Vera C. Rubin Observatory (formerly the Large Synoptic Survey Telescope, or LSST). This system is engineered from the ground up to process, distribute, and archive the unprecedented volume of astronomical data generated by the observatory's telescope and camera systems.

### 1.1 The Project at a Glance

The software and infrastructure to be produced will be officially governed under the *LSSTCorp Data Management* project, and will be referred to as LSST DM or "the system" in the rest of this document.

The primary purpose of LSSTCorp Data Management is to be responsible for creating the software, services, and systems which will be used to produce the Rubin Observatory’s data products. In historical astronomical surveys, data processing was often siloed and executed using ad-hoc scripts on local clusters. For the Rubin Observatory, which will image the entire visible sky every few nights, producing tens of terabytes of data daily, a bespoke, globally distributed processing and database framework is strictly required.

To ensure the entire engineering and scientific body is aligned on this massive undertaking, the leadership team must finalize the official DM mission statement and expose all details and TBDs in all aspects of the DM system design to the public and scientific community.

The high-level goals of this new platform instantiation are:
a. To establish a rigorous taxonomy for astronomical data, mandating a formal initiative to change Source/Object Terminology across the entire codebase to prevent confusion between single-epoch detections (Sources) and multi-epoch physical entities (Objects).
b. To engineer a robust, near-real-time processing pipeline (Level 1) capable of detecting transient events—such as supernovae or near-Earth asteroids—and generating worldwide alerts within 60 seconds of observation.
c. To architect an immensely scalable, distributed relational database, known as Qserv, capable of storing and querying petabytes of catalog data efficiently.
d. To provide a modern, highly documented development environment that embraces modern standards, explicitly requiring the infrastructure to enable C++11 support and migrate all version control to standard Git workflows.
e. To build mathematically rigorous algorithmic pipelines capable of performing advanced image difference analysis, source deblending, and forced photometry on billions of distant galaxies simultaneously.

### 1.2 Definitions, acronyms, and abbreviations.

The following alphabetically sorted definitions may help to better understand this document and the complex astronomical and technical specifications required for the system's implementation:

* *CSS* – Cluster State Service. The system uses a Zookeeper-based CSS (v1) to manage the state and configuration of the thousands of distributed compute nodes.
* *DAC* – Data Access Center. The physical and logical facilities where astronomers and the public will access the data products. The architecture requires a detailed SAT drill-down into Level 3 and DAC operations.
* *DayMOPS / NightMOPS* – Moving Object Processing System. The sub-pipelines responsible for linking transient detections across multiple nights to calculate the orbits of solar system objects (asteroids, comets).
* *Diffim* – Difference Image Analysis. The process of subtracting a template image of the sky from a newly acquired image to detect things that have changed or moved.
* *EUPS* – Extended Unix Product System. The package manager used by the LSST software stack to manage complex, multi-version C++ and Python dependencies. The team must explicitly document basic EUPS functionality for onboarding new developers.
* *Qserv* – The bespoke, distributed SQL database designed by LSST to handle petabyte-scale spatial queries across billions of celestial objects.
* *SAT* – Science Algorithm Task. The core unit of work defining the scientific and algorithmic pipelines (e.g., calibration, deblending, coaddition).
* *SDQA* – Science Data Quality Assessment. The automated processes used to ensure the incoming telemetry and resulting data products are scientifically valid.
* *SUI* – Science User Interface. The web-based portal and API endpoints through which scientists will interact with the data catalogs and images.

## 2. Requirements description.

The description of the intended product is detailed in the extensive following subsections, outlining exactly how the core C++ and Python engines, the image processing pipelines, the distributed database nodes, and the scientific user interfaces must behave during standard operation.

### 2.1 LSST Data Management and its environment.

The following description outlines the intended relationship between the Data Management software, the physical telescope hardware on Cerro Pachón, the base facility, and the primary archive centers:

* Raw pixel data flows from the massive gigapixel camera on the telescope to the base facility via dedicated fiber-optic networks. The architecture team must execute a detailed SAT drill-down into telescope/camera interfaces (data transfer) to ensure the software can handle this ingestion without buffer overflows or latency spikes.
* The ingest system processes the raw data, applies basic formatting, and immediately routes the images to the Level 1 processing clusters for rapid transient detection.
* Concurrently, the data is transmitted to the primary archive centers where the Level 2 pipelines run massive, periodic Data Releases (DR) that recalculate the entire history of the survey.
* Processed images and object catalogs are subsequently ingested into the Qserv distributed database.
* Astronomers and the public access these databases and visualizations via the Science User Interface (SUI) hosted at the Data Access Centers (DAC).

The main inputs to the DM system come from raw camera telemetry, calibration screen flat-fields, and user-submitted database queries. It is the job of the system's execution engine to continuously analyze these inputs, manage massive parallel compute tasks, generate real-time alerts for transient phenomena, and deliver flawlessly calibrated data products to the global astronomical community.

### 2.2 Product functions: The Scientific Pipelines.

The main functional requirements of the system are categorized by their specific architectural domain within the astronomical data processing framework. To ensure the mathematical and physical accuracy of the software, the engineering team is mandated to execute deep "Science Algorithm Task" (SAT) drill-downs for every major pipeline stage.

**Level 1 Processing: Nightly Operations and Alert Generation**
a. The system shall provide a near-real-time processing capability known as Level 1. The architecture demands a comprehensive SAT drill-down into Level 1 calibrated exposure processing, which entails removing instrumental signatures (bias, dark current, flat-fielding) from raw images within seconds of the shutter closing.
b. Following calibration, the pipeline must execute Difference Image Analysis. The team must perform a SAT drill-down into Level 1 diffim and NightMOPS. By subtracting a historical template image from the new exposure, the software will isolate objects that have changed in brightness or position.
c. Immediate dissemination of discoveries is mandatory. The system requires a SAT drill-down into alert gen and distribution. Any detected transient (e.g., a supernova) must be packaged into a standardized alert format and broadcast to worldwide broker networks within 60 seconds.
d. The system must process solar system objects during non-observing hours. The architecture requires a SAT drill-down into daytime calibration (DayMOPS), which links the unassociated transient detections from previous nights to calculate precise orbital trajectories for asteroids and comets.

**Level 2 Processing: Data Releases and Deep Coaddition**
e. The system shall provide a massive, periodic recalculation engine known as Level 2. The team must execute a SAT drill-down into how a data release occurs, orchestrating the reprocessing of petabytes of accumulated historical data to create the most accurate possible static sky maps.
f. To see fainter objects, the software must stack multiple images. The architecture demands a SAT drill-down into deep coadds and coadd detection, mathematically combining hundreds of individual exposures into a single, deep master image.
g. The system must establish the baseline references used by Level 1. This requires a precise SAT drill-down into backgrounds and templates generation, ensuring the reference images used for subtraction are free of artifacts and cosmic rays.
h. As images are stacked, distinct galaxies often overlap. The system must provide mathematically robust algorithms for this; requiring a SAT drill-down into deblending and source association to accurately separate the light profiles of overlapping celestial bodies.

i. Furthermore, the system must refine the existing algorithmic structure by moving existing algorithms from `meas_algorithms` to `meas_base` and refactoring them to the new API, ensuring the core measurement logic is modular and highly optimized.

**Advanced Photometry and Mathematical Optimization**
j. To accurately measure the flux and shape of galaxies across multiple epochs, the system must utilize forced photometry. The architecture requires a SAT drill-down into multifit and forced photometry.
k. To achieve the required computational performance for this fitting, the engineering team must add a derivatives-based optimizer to `meas_multifit`. This optimizer will minimize a $\chi^2$ objective function utilizing Levenberg-Marquardt or similar gradient-descent methodologies. For a modeled galaxy profile parameterized by $\theta$, the optimizer will rely on analytic derivatives to solve:


$$\chi^2(\theta) = \sum_{i} \left( \frac{D_i - M_i(\theta)}{\sigma_i} \right)^2$$


where $D_i$ is the observed pixel data, $M_i(\theta)$ is the model evaluation, and $\sigma_i$ is the pixel variance.

**Calibration and Data Quality**
l. The system must maintain its own physical accuracy. A SAT drill-down into Calibration Products Production is mandated to define how the system will automatically process dome flats, bias frames, and standard star observations into master calibration products.
m. To prevent corrupted data from polluting the archives, the system must implement strict quality gates. The architecture requires a SAT drill-down into archive ingest and SDQA (Science Data Quality Assessment), utilizing machine learning and statistical heuristics to flag bad telemetry automatically.

## 3. Other considerations.

This section provides necessary, non-functional information on what is expected from the system and on characteristics of its environment, including massive database scaling metrics, developer tooling requirements, C++ programming constraints, and the strict release management processes required for an international scientific collaboration.

### 3.1 Reliability, High Availability, and Alert Generation

Reliability of the C++ processing layer, the data transfer network, and the resulting public alerts has a critical priority. The Rubin Observatory costs hundreds of millions of dollars to construct; if the software system drops telescope telemetry or fails to process an exposure, irreplaceable scientific data regarding transient events is lost forever.

The system must guarantee operational stability across thousands of compute nodes. The state of the cluster cannot be managed manually. The architecture team must deploy a robust configuration management system, specifically engineering a Zookeeper-based CSS (v1) (Cluster State Service) to provide distributed synchronization and high-availability failover for the processing nodes.

Furthermore, the data must be highly visual and interpretable for quality control operators. The architecture team must perform a SAT drill-down into visualization to establish the protocols and image compression algorithms necessary to render gigapixel-scale images in web browsers without exhausting local memory.

### 3.2 Information volume and Database Architecture (Qserv)

The system must be engineered to manage the largest astronomical catalog in human history. The final database will contain trillions of rows and require petabytes of storage. Standard relational databases (like standard MySQL or PostgreSQL) cannot scale to this volume or execute the required spherical geometry queries efficiently.

Therefore, the system will utilize the custom-built Qserv distributed database. The database engineering team must formulate the Qserv configuration - detailed design to define how data will be sharded spatially across hundreds of database nodes based on the celestial sphere.

To support the massive, automated expansion of this database cluster, the infrastructure must be entirely scripted. The team must create the Node configuration and bootstrapping - detailed design, and execute the Node bootstrapping - 1st prototype to prove that new database servers can be brought online and integrated into the Qserv cluster automatically. Furthermore, the team must finalize the S14 Configuration for all Qserv components, including node bootstrapping logic for the upcoming milestones.

### 3.3 Developer interfaces, Version Control, and Tooling

The community of astronomers, physicists, and software engineers requires extensive documentation, predictable APIs, and standardized developer environments. The onboarding experience has historically been a bottleneck, and this project instantiation must resolve it.

The technical writing team is strictly mandated to document the software to make it usable to new users and developers. Because the LSST stack relies on the Extended Unix Product System (EUPS) for dependency management, the team must explicitly document basic EUPS functionality.

The source control infrastructure must be modernized to support open-source collaboration. The system administrators must execute a full migration to transition git repositories to Stash (Atlassian Bitbucket) for internal enterprise management, while simultaneously setting up systems to mirror LSST git repositories to Github for public visibility and community contributions. To support agile branch management, the infrastructure team must modify gitolite permissions to allow `issue/DM-NNNN` branches natively.

The installation process must be bulletproof across varying academic environments. The deployment team must update the `newinstall.sh` script to explicitly check for the existence of git and python on users' machines before attempting to compile the stack, providing clear error messages if prerequisites are missing. They must also verify and update URLs in `newinstall.sh` to point to the new, permanent package repositories, and copy `newinstall.sh` to the standard location on the official website.

### 3.4 User characteristics and Scientific Interface

The intended users of the LSST Data Management system span from highly technical C++ algorithm developers to research astronomers querying the database via Python Jupyter notebooks, and eventually to citizen scientists and the general public.

Because the platform will be used by the global scientific community, transparency and communication are paramount. The project management team must ensure they open up LSST software mailing lists to the public. To build a durable knowledge base, the team must transition to Confluence Questions (a Q&A platform) to capture architectural decisions and user support queries. Furthermore, to foster community engagement, the project must actively find DM speakers for LSST Tech Talks to present the pipeline architecture to external institutions.

### 3.5 Programming language and architectural constraints

The primary programming languages utilized for the underlying execution engine of the DM platform are Python (for orchestration, high-level logic, and user interfaces) and C++ (for mathematically intensive image processing and the core database engine).

To ensure the algorithmic pipelines can leverage modern hardware optimizations and standard library features, the architecture team must explicitly enable C++11 support across the entire build system and compiler toolchains.

The deployment and packaging architecture must be meticulously configured to support varied Unix-like operating systems. While Linux is the primary deployment target for the data centers, developers heavily utilize Apple hardware. Therefore, the release engineering team must confirm that the software stack builds on OS X 10.8 natively without requiring complex virtual machines.

### 3.6 Process requirements and Release Management

As a federally funded, massive-scale scientific project, rigorous organizational governance, testing, and release management processes must be fulfilled. The product's success relies entirely on predictable, stable release cycles that scientists can depend on to write their research papers.

The immediate goal for the initial phase is the successful execution of the Winter 2014 release. The release management team must coordinate the I&T (Integration and Testing) and packaging tasks needed to publish the W'14 release. This involves strict version control protocol: the release manager must `git tag` all packages for release, ensuring a reproducible build state.

Simultaneously, the infrastructure team must Release EUPS 1.3.0 to support the new dependencies of the stack. The deployment team must determine the final URL/location for the W'14 stack, explicitly update installation instructions, and actively Build the Winter'14 release binaries. Once verified by QA, the project leadership will officially announce the Winter 2014 release to the community.

Finally, the project relies on intense collaborative design sessions to overcome architectural hurdles. The project management team must organize the S14 Design/hackathon week (scheduled for Apr 28 - May 2 @ SLAC). This logistical requirement mandates that management arrange meeting rooms and proactively come up with the agenda and discussion topics to ensure the engineering teams utilize the in-person time effectively to solve complex architectural problems like the Qserv topology and Level 1 alert routing.

Only through rigorous adherence to these scientific testing matrices, collaborative design processes, and strict architectural constraints can the LSST Data Management system successfully provide a flawless, secure, and performant foundation for the next decade of astronomical discovery.
